# 需求分析框架参考

本文档提供多种需求分析框架的核心要点和应用场景。

## 1. 用户故事 (User Story)

### 格式
```
作为[用户角色]，我想要[目标/需求]，以便[价值/原因]
As a [user role], I want [goal/desire], so that [benefit/value]
```

### 应用要点
- **独立性**: 每个故事应该独立
- **可协商**: 细节可以讨论
- **有价值**: 对用户或业务有明确价值
- **可估算**: 可以估算工作量
- **小**: 足够小，可在一个迭代完成
- **可测试**: 有明确的验收标准

## 2. 用例分析 (Use Case Analysis)

### 核心组成
- **参与者**: 与系统交互的用户或外部系统
- **前置条件**: 用例执行前必须满足的条件
- **主流程**: 正常情况下的步骤序列
- **扩展流程**: 可选路径或分支
- **异常流程**: 错误情况的处理
- **后置条件**: 用例执行后系统的状态

## 3. KANO模型

### 五类需求特性

**必备型需求**: 用户认为理所当然的基本功能
**期望型需求**: 用户明确表达的需求
**魅力型需求**: 用户未明确表达但能带来惊喜的需求
**无差异需求**: 用户不关心的功能
**反向需求**: 某些用户不希望有的功能

## 4. STAR法则

### 框架结构
- **S (Situation)**: 情境 - 用户在什么场景下
- **T (Task)**: 任务 - 用户需要完成什么任务
- **A (Action)**: 行动 - 用户采取什么行动
- **R (Result)**: 结果 - 期望达到什么结果

## 5. MECE原则

### 定义
- **相互独立 (Mutually Exclusive)**: 各部分之间无重叠
- **完全穷尽 (Collectively Exhaustive)**: 所有部分加起来覆盖全部

### 应用领域
- 功能模块划分
- 问题分析
- 流程梳理
- 异常点分析

## 框架选择指南

| 分析目标     | 推荐框架          |
| -------- | ------------- |
| 快速理解用户需求 | 用户故事 + STAR法则 |
| 系统功能设计   | 用例分析          |
| 功能优先级排序  | KANO模型        |
| 功能模块划分   | MECE原则        |
| 全面需求梳理   | 组合使用多个框架      |
